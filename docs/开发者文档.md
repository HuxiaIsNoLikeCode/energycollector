## 4. 核心架构：Storage-First 数据策略

### 4.1 为什么不用 $app 中转？

Vela OS 上 `$app` 引用在某些时机（如页面销毁时）会失效，导致通过 `app.gameState` 写入的数据静默失败。经过多轮迭代验证，最终采用 **storage-first** 策略：

```
所有页面 → 直接 storage.get() / storage.set()
$app 仅作 best-effort 内存同步（成功最好，失败不影响正确性）
```

### 4.2 全局数据流

```
┌─────────┬────────────────┬─────────────────┐
│  页面   │      读        │       写         │
├─────────┼────────────────┼─────────────────┤
│ game    │ storage.get()  │ storage.set()   │
│         │ gameData+appSett│ gameData(异步)  │
│ index   │ storage.get()  │ （只读显示）      │
│         │ gameData       │                 │
│ upgrade │ storage.get()  │ storage.get()+  │
│         │ gameData       │ storage.set()   │
│ settings│ storage.get()  │ storage.set()×2 │
│         │ appSettings    │ gameData+appSett│
└─────────┴────────────────┴─────────────────┘
```

### 4.3 Storage Keys

| Key | 格式 | 内容 | 读写方 |
|-----|------|------|--------|
| `gameData` | JSON | `{ highScore, energy, upgrades }` | game, upgrade, settings(重置) |
| `appSettings` | JSON | `{ vibrateOn, difficulty, particlesOn }` | settings, game(读取) |

### 4.4 game.ux 写入流程（关键）

```javascript
_writeResultToApp() {
  // 1. 防重入守卫（同步，立即生效）
  if (this._saved) return
  this._saved = true

  // 2. 从 storage 读出现有真实值（异步）
  storage.get({
    key: 'gameData',
    success: function(data) {
      var existing = JSON.parse(data)
      var oldEnergy = existing.energy || 0
      var newTotal = oldEnergy + earnedEnergy  // 在真实值上累加！

      // 3. 更新界面显示
      self.totalEnergy = newTotal

      // 4. 保留 storage 中的 upgrades（不被游戏页过期值覆盖）
      var saveData = JSON.stringify({
        highScore: newHigh,
        energy: newTotal,
        upgrades: existing.upgrades || self.upgrades  // 关键！
      })

      // 5. 写回 storage
      storage.set({ key: 'gameData', value: saveData })
    }
  })
}
```

**三个关键设计决策**：
1. `_saved` 守卫在异步调用之前同步设置，防止竞态重复写入
2. 累加基数从 storage 读而非从 `$app` 内存快照读（$app 可能过期）
3. upgrades 字段保留 storage 中的值而非游戏页的本地快照（防止覆盖玩家升级数据）

## 5. 三档难度系统

### 5.1 难度参数矩阵

```javascript
applyDifficulty() {
  if (diff === 0) {  // 简单
    obstacleSpeedMul  = 0.6    // 障碍物慢
    obstacleSpawnRate = 0.6    // 障碍物少
    particleSpawnRate = 1.3    // 粒子多
    particleSizeMul   = 1.2    // 粒子大
    particleValueMul  = 0.8    // 分值低
    collisionRange    = 5      // 碰撞宽松(+5px)
  }
  else if (diff === 1) {  // 困难
    obstacleSpeedMul  = 1.4
    obstacleSpawnRate = 1.3
    particleSpawnRate = 0.7
    particleSizeMul   = 0.85
    particleValueMul  = 1.5
    collisionRange    = -3     // 碰撞严格(-3px)
  }
  else if (diff === 2) {  // 地狱
    obstacleSpeedMul  = 1.8
    obstacleSpawnRate = 1.6
    particleSpawnRate = 0.45
    particleSizeMul   = 0.65
    particleValueMul  = 2.5
    collisionRange    = -6
  }
}
```

### 5.2 各参数如何影响游戏

| 参数 | 影响位置 | 公式 |
|------|----------|------|
| `obstacleSpeedMul` | spawnObstacle() | `speed * obstacleSpeedMul` |
| `obstacleSpawnRate` | update() | `obstacleRate = max(600, baseRate / obstacleSpawnRate)` |
| `particleSpawnRate` | update() | `spawnRate = max(800, baseRate / particleSpawnRate)` |
| `particleSizeMul` | spawnParticle() | `size = round(baseSize * particleSizeMul)` |
| `particleValueMul` | spawnParticle() | `value = round(baseValue * particleValueMul)` |
| `collisionRange` | checkCollisions() | `hitDist = 22 + collisionRange` |

## 8. 页面生命周期

### 8.1 应用生命周期 (app.ux)

```
onCreate()
  → 初始化 data (gameState + settings + listeners)
  → loadFromStorage()
    → storage.get('gameData')  → 解析到 gameState
    → storage.get('appSettings') → 解析到 settings
    → 两个都完成 → _notifyReady()
onDestroy() → 空
```

### 8.2 游戏页面 (game.ux)

```
onInit()
  → loadSettings()           // 异步读 appSettings
  → 注册 onDataReady 回调
  → (dataReady 时) _initFromApp()
    → 读 $app.gameState 到 _baseEnergy/_baseHighScore/upgrades
    → applyUpgrades()
    → startGame()

onReady()
  → keepScreenOn(true)

startGame()  // 注意：在 _initFromApp 回调中调用
  → _gameStarted = true; _saved = false
  → 重置所有游戏状态
  → applyUpgrades(); applyDifficulty()
  → startSensors()           // ★ 传感器在此订阅
  → startLoop()              // ★ 游戏循环在此启动

onDestroy()
  → stopGame(); stopSensors(); keepScreenOn(false)
  → saveProgress()           // best-effort 写入
```

### 8.3 其他页面

```
index.onShow()    → loadData() → storage.get('gameData') → 显示
upgrade.onShow()  → loadData() → storage.get('gameData') → 显示
settings.onShow() → loadSettings() → storage.get('appSettings') → 显示
upgrade.upgrade()→ 扣除能量 → storage.get 保留highScore → storage.set 写回
settings.doReset()→ storage.set({gameData: 初始值})
```

## 9. 已踩过的坑（经验总结）

### 9.1 $app 在 Vela OS 上不可靠

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 能量不累积 | $app.gameState 写入被吞 | 改为直连 storage |
| 升级数据回滚 | 游戏结束用本地快照覆盖了 storage 中的真实升级数据 | 写入时读 existing.upgrades 保留 |
| 设置不保存 | 同上 | settings 也改为 storage 直连 |

### 9.2 异步竞态

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 能量重复累加 | hitObstacle 和 goBack 都调 saveProgress，两次异步回调都执行 | `_saved` 标志位在 `_writeResultToApp()` 入口同步设置 |

### 9.3 传感器订阅时机

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 陀螺仪无效 | onReady 时页面未就绪，subscribe 静默失败 | 移入 startGame() 中调用 |

### 9.4 storage.get/set 是异步的

```javascript
// 正确 ✅：先读再写
storage.get({ key: 'k', success: function(d) {
  var val = JSON.parse(d).energy + earned
  storage.set({ key: 'k', value: JSON.stringify({energy:val}) })
}})

// 错误 ❌：依赖内存快照写
var val = this._baseEnergy + earned  // _baseEnergy 可能过期！
storage.set({ key: 'k', value: ... })
```

### QuickJS 限制备忘

- **ES5 only**: var / function / 匿名 function
- **无 Promise**: 全部用 callback
- **无模块作用域**: import 只能在顶层
- **callback 必须是普通 function**: sensor 的 callback 不能用箭头函数
- **性能**: 50ms 循环间隔是平衡点，不要低于 30ms
